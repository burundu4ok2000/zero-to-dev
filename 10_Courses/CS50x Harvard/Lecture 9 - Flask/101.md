# One‑liner
The built‑in **Flask** server (`flask run`) is **only for development**. For production you must run your app behind a **WSGI** server (e.g., **Gunicorn**, **uWSGI**, **Waitress**)—often with a reverse proxy like **Nginx**.

# Very simple
- **Dev server** ⇒ convenient for local testing: auto‑reloads, debug; **not secure or scalable**.
- **WSGI server** ⇒ speaks the **WSGI** interface to your `app` object, manages **workers/threads**, timeouts, and robustness; designed for **internet traffic**.

# Minimal examples
**Gunicorn (Linux/macOS):**
```bash
pip install gunicorn
gunicorn -w 4 -b 0.0.0.0:8000 "app:app"
```
**Waitress (Windows-friendly):**
```bash
pip install waitress
waitress-serve --port=8000 app:app
```
**uWSGI (another WSGI server):**
```bash
pip install uwsgi
uwsgi --http :8000 --wsgi-file app.py --callable app --processes 4 --threads 2
```

A tiny `wsgi.py` entry point (some hosts expect `application`):
```python
# wsgi.py
from app import app as application
```

# Why not the dev server?
- **Single-process** (or limited), **no hardening**, and can expose debug internals.
- Lacks **production-grade** features (timeouts, graceful restarts, connection handling, security hardening).

# Quick checklist for prod
- Run under a **WSGI server**, keep `debug=False`.
- Put a **reverse proxy** (e.g., Nginx) in front for TLS, gzip, static files, and buffering.
- Configure **logging**, **environment variables**, and a **process supervisor** (systemd, Supervisor) for restarts.
